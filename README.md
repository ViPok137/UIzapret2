# Zapret Launcher

GUI-лаунчер и модуль автообновления для Windows-форка утилиты **zapret**, реализованный
по ТЗ от 16.07.2026. Технологии: **C# / .NET 8, WinForms**.

## Структура репозитория

```
ZapretLauncher/
├── ZapretLauncher.sln
├── README.md
├── .gitignore
└── ZapretLauncher/                  ← проект
    ├── ZapretLauncher.csproj
    ├── app.manifest                 ← манифест UAC (asInvoker, см. ниже почему)
    ├── Program.cs                   ← точка входа, запрос повышения прав
    ├── MainForm.cs                  ← главное окно + трей
    ├── ListEditorForm.cs            ← редактор list.txt
    ├── config.json                  ← пример конфига (копируется рядом с exe при сборке)
    ├── list.txt                     ← пустой файл-заглушка (копируется рядом с exe)
    ├── Models/
    │   └── AppConfig.cs             ← AppConfig, Preset (схема config.json из п.5 ТЗ)
    ├── Services/
    │   ├── ConfigManager.cs         ← загрузка/сохранение config.json
    │   ├── ProcessManager.cs        ← запуск/остановка winws.exe (п.3.2)
    │   ├── UpdateService.cs         ← автообновление через GitHub API (п.3.3)
    │   ├── AutostartManager.cs      ← автозапуск через реестр (HKCU\...\Run)
    │   └── ElevationHelper.cs       ← проверка прав администратора
    └── Resources/
        └── app.ico
```

Этот репозиторий соответствует `my-zapret-launcher` из архитектуры в разделе 2 ТЗ.
Бинарники (`winws.exe`, `WinDivert.dll`, `WinDivert64.sys`) сюда не входят — они
приходят из вашего форка `my-zapret-core` (форк `bol-van/zapret-win-bundle`), как и
описано в ТЗ.

## Сборка и запуск

Нужен **.NET 8 SDK** ([dotnet.microsoft.com](https://dotnet.microsoft.com/download)) и Windows 10/11
(WinDivert — Windows-only драйвер, кроссплатформенности тут не будет).

```powershell
# Обычная сборка (запуск из-под IDE/F5 тоже работает)
dotnet build -c Release

# Публикация в один exe-файл, без установленного рантайма на машине пользователя
dotnet publish ZapretLauncher\ZapretLauncher.csproj -c Release -r win-x64 `
    --self-contained true -p:PublishSingleFile=true -p:IncludeNativeLibrariesForSelfExtract=true
```

Результат `dotnet publish` окажется в
`ZapretLauncher\bin\Release\net8.0-windows\win-x64\publish\`.

> **Важно:** я собирал этот проект в Linux-песочнице без .NET SDK и без доступа в
> интернет, поэтому **не смог прогнать `dotnet build` и проверить компиляцию** —
> сначала соберите у себя и пришлите мне ошибки компиляции, если будут (для WinForms
> с современным .NET SDK их обычно немного и они мелкие).

## Структура папок у собранного приложения (рантайм)

```
ZapretLauncher.exe
config.json          ← настройки, создаётся автоматически при первом запуске, если нет
list.txt             ← список доменов для пресета "Только сайты из списка"
bin/                 ← сюда кладутся winws.exe, WinDivert.dll, WinDivert64.sys
logs/
└── winws.log        ← stdout/stderr процесса winws.exe (для отладки стратегий)
```

Папку `bin/` при первом запуске нужно наполнить вручную (скопировать релиз
`my-zapret-core`) либо один раз нажать «Проверить обновления» — автообновитель создаст
и заполнит её сам.

## Перед первым запуском: обязательно отредактируйте `config.json`

```json
"update_repo_owner": "YOUR_GITHUB_USERNAME",
"update_repo_name": "my-zapret-core"
```

Впишите туда реальные владельца/имя вашего форка — иначе автообновитель обращается в
никуда и просто не находит релиз (это не сломает приложение, просто обновления не
найдутся).

## Ключевые решения и допущения

ТЗ оставляло несколько технических деталей на усмотрение реализации — вот что я выбрал
и почему:

- **WinForms, а не WPF.** В п.3.2 ТЗ явно фигурирует `CreateNoWindow = true` — это
  точное имя свойства `ProcessStartInfo` в .NET, сильный сигнал, что ожидается именно
  .NET-стек. WinForms проще для такого набора фич (трей, несколько простых форм) и не
  требует лишнего кода по сравнению с WPF.
- **«Служба» — это дочерний процесс лаунчера, а не служба Windows (SCM).** В п.3.2
  слово «служба» использовано, но признаков именно `Windows Service` (отдельный
  service-хост, `sc create`, работа без залогиненного пользователя) в ТЗ нет — а
  `CreateNoWindow` явно про обычный процесс. Поэтому `winws.exe` живёт как дочерний
  процесс лаунчера: просто и достаточно для GUI-инструмента. Если позже понадобится,
  чтобы обход работал без запущенного лаунчера или до входа пользователя в систему —
  это отдельная задача (в `zapret-win-bundle` уже есть свои `service*.cmd` на основе
  штатной регистрации службы, можно оттолкнуться от них).
- **Повышение прав — через `runas` в коде, а не `requireAdministrator` в манифесте.**
  С `requireAdministrator` в манифесте Windows при отказе от UAC вообще не запускает
  процесс (ошибка 1223) — приложению нечем поймать этот момент и показать своё
  сообщение, как требует п.3.1. Поэтому манифест — `asInvoker`, а `Program.cs` сам
  пытается перезапуститься с `Verb = "runas"` и ловит отказ пользователя.
- **`list.txt` в репозитории пустой.** Я не нашёл достоверного подтверждения, что
  парсер hostlist в `winws.exe` поддерживает строки-комментарии (`#`) — рисковать не
  стал, чтобы не заспамить ваш будущий список доменов "невалидной" первой строкой.
  Формат (один домен на строку) объясняется в README и прямо в окне редактора.
- **Сравнение версий при автообновлении** — упрощённое: если тег релиза парсится как
  `Major.Minor.Build.Revision`, сравниваются числа; иначе — просто как строки. Для
  тегов вида `v1.4.2` работает; для тегов с суффиксами (`v1.4.2-beta`) считает их
  всегда "другой" версией, что на практике почти всегда означает "есть обновление" —
  для более тонкой семантики потребуется полноценный semver-парсер.
- **Перед установкой обновления лаунчер всегда спрашивает подтверждение**, даже если
  включена настройка «проверять обновления при запуске» — она включает только
  автоматическую *проверку*, не бесшумную установку. Это конфигурация, которая
  останавливает процесс и подменяет бинарники, молча делать это на старте не хотелось.

## Что не реализовано (осознанно, для последующих итераций)

- CRUD-редактор пресетов в GUI (сейчас — выбор из списка, как и требует п.3.4; для
  редактирования состава пресетов используется сам `config.json`).
- Полноценная служба Windows (SCM) вместо дочернего процесса — см. выше.
- Единственный запущенный экземпляр (single instance lock) — с учётом того что при
  первом запуске приложение само себя перезапускает для UAC, эту защиту стоит добавлять
  отдельно и аккуратно, чтобы не столкнуть между собой неповышенный и повышенный
  экземпляры при рестарте.
